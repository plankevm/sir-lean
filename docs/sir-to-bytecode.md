# SIR to Bytecode Correctness

The central Plank question is SIR-to-bytecode, not source-to-SIR. MIR-to-SIR matters later, but the first useful target is a theorem that generated bytecode preserves the SIR semantics.

## Proposed Theorem Shape

```text
compile_sir p = bytecode
initial_rel p sir_state evm_state
run_sir fuel_sir p sir_state = sir_result
---------------------------------------------
exists evm_result,
  run_evm fuel_evm bytecode evm_state = evm_result
  and result_rel sir_result evm_result
```

For single operations or basic blocks, use a local simulation theorem:

```text
emit_op op = bytes
state_rel_before op sir_state evm_state
step_sir_op op sir_state = sir_state'
---------------------------------------------
exists evm_state',
  run_evm_segment bytes evm_state = evm_state'
  and state_rel_after op sir_state' evm_state'
```

## Direct EVM Operations

SIR has operations that map almost directly to EVM opcodes:

```rust
OperationKind::SLoad => op::SLOAD,
OperationKind::SStore => op::SSTORE,
OperationKind::Call => op::CALL,
OperationKind::StaticCall => op::STATICCALL,
OperationKind::Return => op::RETURN,
OperationKind::Revert => op::REVERT,
```

Source: [`forks/plank-monorepo/plankc/sir/crates/data/src/operation/mod.rs`](../forks/plank-monorepo/plankc/sir/crates/data/src/operation/mod.rs)

For these, the proof mostly establishes that operand order, stack layout, gas assumptions, memory slices, and returndata conventions line up.

## SIR-only Operations

Some SIR operations are not one opcode. They are compiler conveniences or stack/memory scheduling operations:

- `SetSmallConst`, `SetLargeConst`;
- `MemoryLoad`, `MemoryStore`;
- static and dynamic allocation operations;
- `InternalCall`;
- code/data offset and length operations.

The release backend exposes that boundary:

```rust
Operation::MemoryLoad(data) => self.emit_memory_load(data),
Operation::MemoryStore(data) => self.emit_memory_store(data),
Operation::SetSmallConst(args) => self.asm.push_minimal_u32(args.value),
Operation::InternalCall(args) => {
    self.emit_icall(state, icall_return_marks, op_idx, args.function)
}
```

Source: [`forks/plank-monorepo/plankc/sir/crates/release-backend/src/code_to_asm.rs`](../forks/plank-monorepo/plankc/sir/crates/release-backend/src/code_to_asm.rs)

For these, prove correctness against emitted instruction sequences, not one-step opcode semantics.

## What the State Relation Must Say

At minimum:

- SIR locals are represented by a stack layout, memory spill layout, or both.
- SIR current block/instruction corresponds to an EVM byte offset/program counter.
- SIR memory abstraction matches the EVM frame memory for the relevant slices.
- SIR storage/transient storage match the account storage maps.
- SIR call context matches EVM calldata/caller/value/static/depth fields.
- SIR returndata and logs match EVM returndata/logs.
- SIR halt/revert/exception outcomes match EVM `RETURN`, `REVERT`, and exceptional halt outcomes.

Vyper-HOL's Venom codegen architecture is the closest precedent. It separates register/stack/memory relations, full state relations, and terminal observable relations. That split is more scalable than one giant equality theorem.

## Why This Should Not Start at Raw EVMYulLean

If SIR proofs target EVMYulLean directly from day one, every theorem inherits bytecode program counters, jump destinations, gas errors, parsing, stack height, and memory expansion. That is the correct target eventually, but a poor first semantic interface.

The recommended staging is:

1. executable SIR semantics;
2. backend emission lemmas for simple operations;
3. stack/memory layout relation;
4. block-level simulation;
5. full function/control-flow simulation;
6. whole-runtime bytecode theorem.
